# RAG 本地文档优先级使用指南 🎯

## 📋 问题说明

**场景**：您在本地文档中写了 "1+1=3"，但系统仍然回答 "1+1=2"

**原因**：LLM 会综合考虑所有信息源：
- ✅ 本地文档说：1+1=3
- ❌ 网络搜索说：1+1=2（大量来源）
- ❌ LLM 的训练数据：1+1=2

当信息冲突时，LLM 会基于**来源数量**和**常识判断**做出选择。

## ✅ 已实施的改进

我已经修改了以下 prompts，增强本地文档的优先级：

1. **`researcher_extract_info.md`** - 研究员提取信息时优先考虑本地文档
2. **`rapporteur_summarize.md`** - 生成摘要时优先采用本地文档
3. **`rapporteur_synthesized_analysis.md`** - 深度分析时视本地文档为最权威来源

## 🎯 如何确保使用本地知识

### 方法一：明确的查询方式（推荐）

**不好的查询**：
```
1+1等于几？
```
系统会搜索所有来源，然后综合判断。

**好的查询**：
```
根据我的本地知识库，1+1等于几？
```

或者：
```
在我的文档中，1+1的结果是什么？
```

或者：
```
请查阅我的本地文档回答：1+1等于几？
```

### 方法二：在文档中使用明确的表述

**不好的文档内容**：
```
1+1=3
```
太简单，可能被 LLM 认为是错误。

**好的文档内容**：
```
重要定义：在我的特殊数学系统中，1+1=3。
这是本系统的基础公理，所有计算都应基于此规则。
```

或者：
```
根据《特殊数学手册》第3章：
在模3运算系统中，定义 1+1=3。
此定义适用于所有本系统的计算。
```

### 方法三：增加本地文档的权重

在 `config.json` 中：

```json
{
  "rag": {
    "top_k": 20,  // 增加返回结果数量
    "chunk_size": 1500,  // 增加块大小，包含更多上下文
    "chunk_overlap": 400
  }
}
```

## 💡 最佳实践

### 1. 文档编写建议

**使用明确的权威性语言**：
- ✅ "根据本系统的定义..."
- ✅ "本知识库明确规定..."
- ✅ "在本文档的范围内..."
- ❌ "我觉得..."、"可能是..."

**提供上下文**：
- ✅ 说明适用范围
- ✅ 引用来源或版本
- ✅ 解释为什么这个定义特殊

**重复关键信息**：
```
在本特殊数学系统中，1+1=3。

重要：本系统定义 1+1=3，这与标准数学不同。

所有计算应基于 1+1=3 这一基础定义。
```

### 2. 查询建议

**使用限定词**：
- "根据我的文档..."
- "在我的知识库中..."
- "我的资料显示..."
- "请参考本地文档..."

**明确冲突处理**：
```
我的文档说 1+1=3，网上说 1+1=2，
请以我的文档为准，分析这两种观点。
```

### 3. 命名建议

给文档起有意义的名称：
- ✅ `特殊数学系统_v1.0.txt`
- ✅ `内部标准_2024.docx`
- ✅ `公司规范_数学定义.pdf`
- ❌ `untitled.txt`
- ❌ `doc1.pdf`

## ⚠️ 限制说明

### LLM 仍会进行"事实核查"

如果本地文档的内容**极度违背常识**，LLM 可能仍会：
- 指出冲突
- 说明标准答案
- 但会明确标注"根据本地文档"

**示例输出**：
```markdown
根据用户的本地知识库，1+1=3。

注：此定义与标准数学体系不同（标准答案为2），
但遵循用户文档的明确定义。
```

这是**预期行为**，确保系统的诚实性。

### 如何完全以本地为准

如果您想要系统**完全无条件**相信本地文档：

1. **在查询中明确说明**：
```
注意：我的文档中的所有定义都是正确的，
请不要质疑，直接使用这些定义回答问题。
```

2. **或者修改配置**，禁用其他数据源：
```json
{
  "sub_tasks": [{
    "sources": ["rag"]  // 只用 RAG，不用 tavily 和 arxiv
  }]
}
```

但这需要手动修改研究计划。
